# 通用属性代理系统 — 方案设计

> 三层架构：**逻辑属性**（来源累加）→ **转换映射**（YAML 配置）→ **物理属性**（推送 AttributeDriver）
> 日期：2026-05-31

---

## 一、设计动机

现有链路的问题：

```
Job.hook.attributes → 字符串列表 → AttributeProvider 合并 → AttributeDriver 推送
```

所有属性直接以最终字符串形式写死在 YAML，例如 `"ATK: +100"`。这意味着：
- 力量加了多少？不知道——只能人肉反算
- 力量转攻击的比例是多少？散落在各处写死，要改得全部改
- 装备/Buff 加的是"攻击"还是"力量"？如果是力量，怎么转成攻击？没有统一入口

**正确做法**：加一层转换。

```
 来源层（逻辑属性）          转换层（YAML 配置映射）        推送层（物理属性）
 ──────────────────         ────────────────────        ──────────────────
 STR: 25              →     STR * 1.0 → ATK  +25       ATK:      +25
                             STR * 0.5 → DEF  +12       DEF:      +12
 INT: 5               →     INT * 1.0 → M_ATK +5        MAGIC_ATK: +5
                             INT * 0.3 → M_REGEN +1      MANA_REGEN:+1
 AGI: 8               →     AGI * 0.8 → SPEED +6        SPEED:    +6
                             AGI * 0.3 → CRIT  +2        CRIT:     +2
```

- **来源层**只管"我有多少力量"
- **转换层**定义"力量能变成什么、系数多少"
- **推送层**只管"把最终数值同步到 AttributePlus"

## 二、三层架构总览

```
                      ┌─────────────────────────────┐
                      │       AttributeProxy         │
                      │   (注册来源 + 逻辑属性汇总)     │
                      └─────────────┬───────────────┘
                                    │ 逻辑属性 Map<String, Double>
                                    │ { STR:25, INT:5, AGI:8, HP:580, ... }
                                    ▼
                      ┌─────────────────────────────┐
                      │    AttributeConversion       │
                      │   (逻辑属性 → 物理属性 映射)    │
                      │   读 Planners.attributeRegistry│
                      └─────────────┬───────────────┘
                                    │ 物理属性 Map<String, Double>
                                    │ { ATK:25, DEF:12, M_ATK:5, SPEED:6, ... }
                                    ▼
                      ┌─────────────────────────────┐
                      │      AttributeDriver.set()    │
                      │   (同步到外部属性插件)          │
                      └─────────────────────────────┘
```

## 三、第一层：逻辑属性来源 AttributeSource

### 3.1 接口

```kotlin
// api/attribute/AttributeSource.kt

interface AttributeSource {
    val id: String
    val priority: Int

    /** 返回该来源贡献的逻辑属性 */
    fun getAttributes(entity: LivingEntity): Map<String, Double>

    companion object {
        const val PRIORITY_BASE     = 0
        const val PRIORITY_SKILL    = 5
        const val PRIORITY_GROWTH   = 10
        const val PRIORITY_INVESTED = 20
        const val PRIORITY_EQUIP    = 30
        const val PRIORITY_BUFF     = 40
    }
}
```

### 3.2 逻辑属性键约定

这些键是**游戏设计语言**，不直接暴露给外部属性插件：

| 类别 | 键 | 含义 |
|------|-----|------|
| 主属性 | `STR` `INT` `AGI` `END` | 力量/智力/敏捷/耐力 |
| 子属性 | `CRIT` `STRIKE` `FAITH` `HASTE` `TOUGH` `SPIRIT` | 暴击/强击/信仰/急速/坚韧/精神 |
| 资源基数 | `HP` `MP` `SP` | HP/MP/SP 的基数部分（用于公式计算） |

- 逻辑属性**不推送到 AttributeDriver**
- 来源只管提供逻辑属性，不管它们最终变成什么

### 3.3 内置来源

不内置任何来源。后续模块实现 `AttributeSource` 接口并 `AttributeProxy.register()` 即可接入。

## 四、第二层：转换映射 AttributeConversion

### 4.1 核心概念

转换层读 YAML 配置，把逻辑属性 Map 换算为物理属性 Map。

```
输入:  { STR: 25, INT: 5, AGI: 8, END: 12, HP: 580, MP: 120, SP: 100, ... }
规则:  STR → { ATK: 1.0, DEF: 0.5 }
       INT → { MAGIC_ATK: 1.0, MANA_REGEN: 0.3 }
       AGI → { SPEED: 0.8, CRIT: 0.3 }
       END → { MAX_HEALTH: 12, TOUGH: 0.5 }
       HP  → { MAX_HEALTH: 1.0 }
       SP  → { MAX_STAMINA: 1.0 }
       ...

输出:  { ATK: 25, DEF: 12, MAGIC_ATK: 5, MANA_REGEN: 1, SPEED: 6, CRIT: 2,
         MAX_HEALTH: 724, TOUGH: 6, MAX_STAMINA: 100, ... }
```

### 4.2 转换配置（config.yml → settings.attribute.registry）

转换规则全局通用，定义在 config.yml 中，与 `settings.condition` 同级：

```yaml
# config.yml
settings:
  attribute:
    registry:
      STR:
        name: "力量"
        mappings:
          ATK: 1.0          # 1 力量 = 1 攻击
          DEF: 0.5          # 1 力量 = 0.5 防御
      INT:
        name: "智力"
        mappings:
          MAGIC_ATK: 1.0    # 1 智力 = 1 法术攻击
          MANA_REGEN: 0.3   # 1 智力 = 0.3 回蓝
      AGI:
        name: "敏捷"
        mappings:
          SPEED: 0.8        # 1 敏捷 = 0.8 速度
          CRIT: 0.3         # 1 敏捷 = 0.3 暴击

        name: "耐力"

          TOUGH: 0.5        # 1 耐力 = 0.5 坚韧

      # 子属性 1:1 透传

        name: "暴击"

          CRIT: 1.0
      STRIKE:
        name: "强击"
        mappings:
          STRIKE: 1.0

        name: "信仰"
```

### 4.3 数据类 + Planners 加载

与 `settings.condition` 相同的模式——`@ConfigNode` + `configNodeToMap` 加载到 `Planners`：

```kotlin
// 数据类
data class AttributeRegistryEntry(
    val key: String,                   // 逻辑属性键，如 "STR"
    val name: String,                  // 展示名，如 "力量"
    val mappings: Map<String, Double>  // 物理键 → 系数
)

// Planners.kt（与 conditions 同级）
@ConfigNode("settings.attribute.registry")
val attributeRegistry = configNodeToMap { key, value ->
    val cfg = value as ConfigurationSection
    val name = cfg.getString("name") ?: key
    val mappingsSection = cfg.getConfigurationSection("mappings")
    val mappings: Map<String, Double> = if (mappingsSection != null) {
        mappingsSection.getValues(false).mapValues { (it.value as Number).toDouble() }
    } else {
        emptyMap()
    }
    AttributeRegistryEntry(key, name, mappings)
}
```

### 4.4 转换引擎

`AttributeConversion` 不自己加载配置，只从 `Planners.attributeRegistry` 读取：

```kotlin
// core/attribute/AttributeConversion.kt

object AttributeConversion {

    /** 全局转换表，运行时从 Planners.attributeRegistry 读取 */
    private val table: Map<String, AttributeRegistryEntry>
        get() = Planners.attributeRegistry.get()

    fun convert(logical: Map<String, Double>): Map<String, Double> {
        val result = mutableMapOf<String, Double>()
        for ((logicalKey, logicalValue) in logical) {
            val entry = table[logicalKey] ?: continue
            for ((physicalKey, factor) in entry.mappings) {
                result[physicalKey] = (result[physicalKey] ?: 0.0) + logicalValue * factor
            }
        }
        return result
    }
}
```

职业差异不体现在转换系数，而体现在**来源层**：战士 growth 给 STR 多、法师 growth 给 INT 多。同一个 `STR: 25`，转换结果永远一致。

## 五、第三层：推送 AttributeDriver

### 5.1 AttributeProxy.sync()

```kotlin
// core/attribute/AttributeProxy.kt

object AttributeProxy {

    private val sources = mutableListOf<AttributeSource>()

    fun register(source: AttributeSource)
    fun unregister(source: AttributeSource)

    /** 获取逻辑属性汇总（所有来源累加） */
    fun getLogical(entity: LivingEntity): Map<String, Double> {
        val result = mutableMapOf<String, Double>()
        for (source in sources.sortedBy { it.priority }) {
            for ((key, value) in source.getAttributes(entity)) {
                result[key] = (result[key] ?: 0.0) + value
            }
        }
        return result
    }

    /** 获取单个逻辑属性 */
    fun get(entity: LivingEntity, key: String): Double {
        return getLogical(entity)[key] ?: 0.0
    }

    /** 重算并推送到外部属性插件 */
    fun sync(entity: LivingEntity) {
        // ① 收集：各来源返回的属性按 key 是否在 registry 中分流
        var logical = mutableMapOf<String, Double>()
        var physical = mutableMapOf<String, Double>()
        val registry = Planners.attributeRegistry.get()
        for (source in sources.sortedBy { it.priority }) {
            for ((key, value) in source.getAttributes(entity)) {
                if (registry.containsKey(key)) {
                    logical[key] = (logical[key] ?: 0.0) + value   // 逻辑属性 → 等转换
                } else {
                    physical[key] = (physical[key] ?: 0.0) + value  // 物理属性 → 直接合并
                }
            }
        }
        // ② 转换：逻辑 → 物理
        val converted = AttributeConversion.convert(logical)
        // ③ 合并：物理直接量 + 转换结果
        for ((key, value) in converted) {
            physical[key] = (physical[key] ?: 0.0) + value
        }
        // ④ 推送
        val list = physical.map { (k, v) -> "$k: +${v.toInt()}" }
        AttributeDriver.set(entity, "planners-proxy", list, -1)
    }
}

/** 获取单个逻辑属性（仅注册表中的 key 参与汇总） */
fun get(entity: LivingEntity, key: String): Double {
    if (!Planners.attributeRegistry.get().containsKey(key)) return 0.0
    var total = 0.0
    for (source in sources.sortedBy { it.priority }) {
        val v = source.getAttributes(entity)[key] ?: continue
        total += v
    }
    return total
}
```

### 5.2 完整数据流

```
玩家升级 (Lv 1 → 2)
  │
  ▼
AttributeProxy.sync(player):
  │
  │ ① 收集 + 分流
  │    registry 中有 STR/INT/AGI/END/HP/MP/SP/CRIT/... → 走逻辑管线
  │    registry 中无 ATK/DEF/MAX_HEALTH/...          → 走物理直通
  │
  ├─ (来源A: 逻辑属性)  → STR:10, INT:2, HP:300, ...        (逻辑)
  └─ (来源B: 物理属性)  → ATK:15                              (物理，直接量)
  │
  │ ② 转换
  │  logical = { STR:10, INT:2, END:8, HP:300, ... }
  │  ──AttributeConversion.convert()──
  │  converted = { ATK:10, DEF:5, MAGIC_ATK:2, MAX_HEALTH:396, ... }
  │
  │ ③ 合并: physical + converted → { ATK:25, DEF:5, MAX_HEALTH:396, ... }
  │
  │ ④ 推送
  └─ AttributeDriver.set(player, "planners-proxy", ["ATK: +25", "DEF: +5", ...], -1)
```

## 六、后续扩展

任何模块实现 `AttributeSource` 接口并注册即可接入属性管线：

```kotlin
class XxxSource : AttributeSource {
    override val id = "xxx"
    override val priority = AttributeSource.PRIORITY_EQUIP
    override fun getAttributes(entity: LivingEntity): Map<String, Double> {
        // 返回逻辑属性或物理属性
    }
}
AttributeProxy.register(XxxSource())
```
